# Trainer Business Logic

## 1. Назначение документа

Этот документ фиксирует текущую бизнес-логику приложения `Trainer` в локальном web-режиме.

Документ нужен как опорная спецификация для:

- понимания пользовательского поведения приложения;
- генерации тестов;
- проверки регрессий при последующих изменениях;
- синхронизации Android- и Web-версий по смыслу, а не по UI-деталям.

Документ описывает именно наблюдаемое поведение приложения и правила обработки данных.
Он не является технической документацией по реализации backend.

## 2. Текущий режим работы

На текущем этапе web-версия приложения работает в изолированном локальном режиме:

- без реального backend;
- без записи тренировок на сервер;
- с загрузкой справочника упражнений и истории тренировок из локальных JSON-фикстур;
- с сохранением пользовательских изменений в `localStorage`.

Это означает, что для пользователя приложение выглядит рабочим, но все созданные им данные живут только локально в браузере.

## 3. Основные сущности

### 3.1. Упражнение

Упражнение в справочнике содержит:

- `id` — числовой идентификатор упражнения;
- `name` — отображаемое название упражнения.

### 3.2. Подход

Подход содержит:

- `set_index` — порядковый номер подхода внутри упражнения;
- `reps` — количество повторений;
- `weight` — рабочий вес;
- `notes` — опциональная заметка.

### 3.3. Упражнение внутри тренировки

Упражнение внутри тренировки содержит:

- `exercise_id`;
- `name`;
- список подходов `sets`.

### 3.4. Тренировка

Тренировка содержит:

- `id` — идентификатор тренировки;
- `workout_date` — дата тренировки в формате `YYYY-MM-DD`;
- `plan_id` — сейчас всегда `null`;
- `data.focus` — сейчас всегда `null`;
- `data.notes` — сейчас всегда `null`;
- `data.load_type` — тип нагрузки;
- `data.exercises` — список упражнений с подходами.

### 3.5. Тип нагрузки

По контракту данных допустимы значения:

- `heavy`
- `medium`
- `light`
- `deload`

В текущем локальном режиме:

- фикстуры используют в основном `heavy`, `medium`, `light`;
- новая локально созданная тренировка получает `load_type` автоматически;
- если в карточке тренировки `load_type` отсутствует, бейдж нагрузки не отображается;
- `deload` допустим по контракту, но в текущем web UI не имеет отдельного особого оформления и попадает в дефолтное отображение.

## 4. Источники данных

Приложение использует два источника данных одновременно.

### 4.1. Фикстуры

Статические JSON-файлы:

- `telegram_miniapp/web/data/exercises.json`
- `telegram_miniapp/web/data/workouts.json`

Они задают базовый справочник упражнений и базовую историю тренировок.

### 4.2. Локальный пользовательский кэш

Созданные пользователем тренировки сохраняются в `localStorage`.

Они не заменяют фикстуры, а добавляются к ним.

### 4.3. Объединение данных

Во всех экранах приложение работает не с одним, а с объединенным списком тренировок:

- сначала берутся локально сохраненные пользовательские тренировки;
- затем добавляются тренировки из фикстур;
- итоговый список сортируется.

### 4.4. Правило сортировки тренировок

Тренировки сортируются:

1. по `workout_date` по убыванию;
2. если дата совпадает, по `id` по убыванию.

Это правило является единым для:

- списка тренировок;
- поиска предыдущих рабочих весов;
- расчета прогресса.

## 5. Локальное хранение состояния

Приложение хранит часть состояния в `localStorage`.

### 5.1. Сохраняемые ключи

- `trainer-miniapp-custom-workouts-v1` — локально созданные тренировки;
- `trainer-miniapp-draft-v1` — черновик текущей незавершенной тренировки;
- `trainer-miniapp-tab-v1` — последняя открытая вкладка;
- `trainer-miniapp-range-v1` — выбранный диапазон на вкладке `Progress`.

### 5.2. Что это означает бизнесово

- пользователь может перезагрузить страницу и не потерять черновик тренировки;
- локально сохраненные тренировки продолжают отображаться после перезагрузки страницы;
- приложение помнит, какая вкладка была активна последней;
- приложение помнит выбранный временной диапазон прогресса.

## 6. Структура приложения

Приложение состоит из трех основных вкладок:

- `Trainings`
- `Progress`
- `New`

По умолчанию при первом запуске web-версии открывается вкладка `New`.

## 7. Вкладка Trainings

### 7.1. Назначение

Вкладка показывает историю тренировок.

### 7.2. Что отображается в карточке тренировки

Каждая тренировка содержит:

- полную дату тренировки;
- день недели;
- бейдж нагрузки, если `load_type` заполнен;
- список упражнений;
- внутри каждого упражнения — список сгруппированных подходов.

### 7.3. Формат даты

Дата отображается в длинном формате на русском языке.

Пример:

- `28 марта 2026`

День недели отображается отдельной строкой и начинается с заглавной буквы.

Пример:

- `Суббота`

### 7.4. Отображение бейджа нагрузки

Если у тренировки есть `load_type`, он показывается как верхний бейдж:

- `LIGHT`
- `MEDIUM`
- `HEAVY`
- либо другое пришедшее значение в верхнем регистре

### 7.5. Логика группировки подходов в истории

Подходы внутри упражнения группируются только если они идут подряд и совпадают по:

- `weight`;
- `reps`;
- `notes`.

Если между одинаковыми подходами встречается другой подход, создаются разные группы.

### 7.6. Правило подписи сгруппированного подхода

Если в группе один подход:

- отображается `Вес кг × Повторения`

Пример:

- `80 кг × 8`

Если в группе несколько подряд идущих одинаковых подходов:

- отображается `Вес кг × Повторения × Количество`

Пример:

- `80 кг × 8 × 3`

### 7.7. Отображение заметок

Если у группы есть `notes`, заметка отображается справа от строки подхода.

### 7.8. Пустое состояние

Если тренировок нет совсем, показывается пустое состояние с сообщением, что пока нет тренировок.

## 8. Вкладка Progress

### 8.1. Назначение

Вкладка показывает агрегированную статистику по объединенной истории тренировок.

### 8.2. Диапазоны анализа

Доступны три диапазона:

- `7D`
- `30D`
- `All`

### 8.3. Логика фильтра по диапазону

Для диапазонов `7D` и `30D` используется включительный интервал:

- верхняя граница — сегодняшняя локальная дата;
- нижняя граница — `today - (days - 1)`.

Это означает:

- `7D` включает сегодня и еще 6 предыдущих дней;
- `30D` включает сегодня и еще 29 предыдущих дней;
- `All` включает все тренировки без фильтра по дате.

### 8.4. Метрики Progress

На вкладке рассчитываются:

- `Workouts` — количество тренировок в выбранном диапазоне;
- `Volume` — суммарный объем;
- `Average Volume / Workout` — средний объем на тренировку;
- `Top Exercise` — упражнение с максимальным суммарным объемом;
- `Heaviest Set` — самый тяжелый подход;
- `Recent Volume Trend` — тренд объема по последним тренировкам.

### 8.5. Что считается объемом

Объем одного подхода:

- `weight * reps`

Подход участвует в объеме только если:

- `weight > 0`
- `reps > 0`

Объем тренировки:

- сумма объемов всех валидных подходов во всех упражнениях этой тренировки.

Общий объем в диапазоне:

- сумма объемов всех валидных подходов во всех тренировках диапазона.

### 8.6. Единица отображения объема

Несмотря на то, что математически это произведение веса на повторения, в UI объем отображается строкой вида:

- `1234 kg`

Для тестов это нужно считать именно текущим пользовательским форматом отображения.

### 8.7. Логика Top Exercise

Для каждого упражнения с одинаковым `name` суммируется объем всех его подходов в диапазоне.

Побеждает упражнение с максимальным суммарным объемом.

Если тренировок нет, `Top Exercise` не отображается.

### 8.8. Логика Heaviest Set

Самым тяжелым считается подход:

1. с максимальным `weight`;
2. при равном `weight` — с максимальным `reps`.

Если тренировок нет, `Heaviest Set` не отображается.

### 8.9. Логика Recent Volume Trend

Для тренда:

- берутся тренировки выбранного диапазона;
- они сортируются от новых к старым;
- затем выбираются максимум 7 последних тренировок;
- затем список разворачивается в порядок от более старой к более новой внутри этих 7 записей.

Итог:

- в тренде всегда отображаются не более 7 последних тренировок диапазона;
- визуально они идут слева направо по времени, от старой к новой.

### 8.10. Поведение при отсутствии данных

Если в диапазоне нет тренировок:

- числовые метрики становятся нулевыми;
- `Top Exercise` и `Heaviest Set` не показываются;
- вместо тренда показывается сообщение `No workouts in this range`.

### 8.11. Обновление данных

На вкладке `Progress` есть ручное обновление.

Оно:

- перечитывает JSON-фикстуры;
- не удаляет локально сохраненные тренировки;
- не удаляет черновик тренировки;
- пересчитывает прогресс на основании обновленного объединенного набора данных.

## 9. Вкладка New

### 9.1. Назначение

Вкладка предназначена для создания новой тренировки.

### 9.2. Базовый сценарий

Пользователь:

1. выбирает упражнение;
2. добавляет к нему подходы;
3. при необходимости переходит к следующему упражнению;
4. завершает тренировку;
5. получает локально сохраненную тренировку, которая сразу появляется в истории и прогрессе.

### 9.3. Автоматическое открытие выбора упражнения

Если:

- справочник упражнений уже загружен;
- текущий черновик пуст;
- упражнение еще не выбрано;

приложение автоматически открывает блок выбора упражнения.

Это поведение нужно считать штатным.

### 9.4. Ограничение на повторное добавление упражнения

В рамках одной тренировки одно и то же упражнение можно добавить только один раз.

Поэтому список доступных упражнений:

- всегда исключает упражнения, уже добавленные в текущую тренировку.

### 9.5. Состояние выбранного упражнения

После выбора упражнения оно становится активным.

Если у него еще нет подходов, пользователь видит карточку:

- название упражнения;
- подпись `Сетов пока нет`.

### 9.6. Доступные действия для активного упражнения

Если упражнение выбрано и пользователь не находится в режиме выбора упражнения или в модальном окне сета, доступны:

- `Добавить сет`
- `Стандарт` — только если для этого упражнения есть валидная история прошлых тренировок

### 9.7. Что считается валидной историей для кнопки Стандарт

Кнопка `Стандарт` доступна, если в истории хотя бы один подход этого упражнения удовлетворяет условиям:

- `weight > 0`
- `reps > 0`

### 9.8. Логика кнопки Добавить сет

При открытии окна добавления сета:

- `reps` по умолчанию устанавливается в `12`;
- `weight` по умолчанию берется из последней подходящей тренировки для этого упражнения.

### 9.9. Как выбирается стартовый вес

Берется история тренировок, отсортированная от новых к старым.

Для выбранного упражнения ищется первая тренировка, где это упражнение присутствует.

Из нее берется:

- максимальный `weight` среди всех подходов этого упражнения.

Если подходящий вес не найден:

- стартовый вес равен `0`.

### 9.10. Логика модального окна сета

Пользователь может менять:

- вес шагом `2.5`;
- повторения шагом `1`.

Ограничения:

- вес не может стать меньше `0`;
- повторения не могут стать меньше `1`.

Доступны действия:

- `Применить`
- `Отмена`

Также закрытие по `Escape` работает как отмена.

### 9.11. Логика кнопки Стандарт

Кнопка `Стандарт` сразу добавляет подход без открытия модального окна.

Параметры такого подхода:

- `reps = 12`
- `weight = weightFromLastWorkout`
- `notes = null`

### 9.12. Добавление подхода в упражнение

Если выбранное упражнение еще не добавлено в текущую тренировку:

- создается новая запись упражнения в черновике.

Если уже добавлено:

- новый подход дописывается в конец списка подходов этого упражнения.

### 9.13. Группировка подходов на экране New

На экране `New` используется отдельная логика группировки:

- подходы группируются только по `weight` и `reps`;
- `notes` в этой группировке не участвуют;
- группировка также работает по подряд идущим подходам.

Формат отображения:

- один подход: `Вес кг × Повторения`
- несколько подряд одинаковых подходов: `Вес кг × Повторения × Количество`

### 9.14. Кнопка Новое упражнение

После того как у текущего выбранного упражнения появился хотя бы один подход, пользователь может нажать `Новое упражнение`.

Это действие:

- не удаляет уже выбранное упражнение из тренировки;
- переводит интерфейс в режим выбора следующего упражнения;
- позволяет выбрать только упражнения, которых еще нет в текущей тренировке.

### 9.15. Отмена выбора нового упражнения

Если пользователь открыл выбор упражнения после уже существующего упражнения и нажал `Отмена`:

- текущий черновик не теряется;
- текущее выбранное упражнение остается активным.

### 9.16. Кнопка Добавить упражнение

Если:

- в тренировке уже есть хотя бы одно упражнение;
- активное упражнение не выбрано;
- есть еще доступные упражнения;

показывается отдельная кнопка `Добавить упражнение`.

### 9.17. Состояние Все упражнения уже добавлены

Если в текущую тренировку уже добавлены все упражнения из локального справочника:

- выбор новых упражнений больше не предлагается;
- пользователь может только завершить тренировку или работать с уже выбранными данными.

### 9.18. Завершение тренировки без данных

Если пользователь нажимает `Закончить тренировку`, а в черновике нет ни одного упражнения:

- тренировка не создается;
- черновик очищается;
- интерфейс возвращается в исходное состояние новой тренировки.

### 9.19. Завершение тренировки с данными

Если в черновике есть упражнения:

- создается новая локальная тренировка;
- она сохраняется в `localStorage`;
- черновик очищается;
- показывается временное сообщение `Тренировка сохранена локально`.

### 9.20. Правила построения новой локальной тренировки

При сохранении формируется объект тренировки со следующими правилами:

- `id = floor(Date.now() / 1000)`
- `workout_date = текущая локальная дата`
- `plan_id = null`
- `focus = null`
- `notes = null`
- `sets[].set_index` начинается с `1` и идет последовательно по порядку добавления

### 9.21. Автоматическое определение load_type

После завершения тренировки `load_type` определяется автоматически по суммарному объему новой тренировки:

- `heavy`, если объем `>= 3000`
- `medium`, если объем `>= 1600` и `< 3000`
- `light`, если объем `< 1600`

## 10. Поведение после сохранения тренировки

После локального сохранения тренировки она должна:

- появиться в `Trainings`;
- участвовать в расчетах `Progress`;
- учитываться как история для кнопки `Стандарт`;
- учитываться как история для определения стартового веса при добавлении сета.

Это ключевая бизнес-инварианта локального режима.

## 11. Правила отображения чисел

### 11.1. Вес

Вес отображается в формате:

- без дробной части, если число целое;
- с одной цифрой после запятой, если есть половинный шаг.

Примеры:

- `80 кг`
- `82,5 кг`

### 11.2. Повторения

Повторения отображаются целым числом.

### 11.3. Объем

Объем отображается как целая часть значения:

- дробная часть отбрасывается;
- затем добавляется суффикс `kg`.

Пример:

- `1547 kg`

## 12. Состояния загрузки и ошибки

### 12.1. Стартовая загрузка

На старте приложение:

- читает локальный черновик и локальные сохраненные тренировки;
- затем загружает фикстуры;
- затем строит итоговое состояние.

### 12.2. Ошибка загрузки фикстур

Если JSON-фикстуры не загрузились:

- приложение показывает экран ошибки;
- на экране есть кнопка повтора.

### 12.3. Повтор после ошибки

Повтор должен:

- снова попытаться прочитать локальные фикстуры;
- восстановить обычную работу приложения, если ошибка исчезла.

## 13. Бизнес-инварианты для тестов

Ниже перечислены инварианты, которые желательно использовать в автотестах.

### 13.1. Инварианты по данным

- локально сохраненная тренировка не удаляет фикстурные тренировки;
- объединенный список тренировок всегда отсортирован по дате и затем по `id`;
- черновик тренировки переживает перезагрузку страницы;
- локально сохраненные тренировки переживают перезагрузку страницы.

### 13.2. Инварианты по вкладке Trainings

- каждая тренировка отображает дату, день недели и упражнения;
- одинаковые подряд идущие подходы в истории объединяются;
- если у группы подходов есть `notes`, они отображаются рядом со строкой подхода.

### 13.3. Инварианты по вкладке Progress

- `Workouts` равно числу тренировок в выбранном диапазоне;
- `Volume` равно сумме `weight * reps` по всем валидным подходам диапазона;
- `Average Volume / Workout = Volume / Workouts`, если `Workouts > 0`;
- `Top Exercise` определяется по максимальному суммарному объему;
- `Heaviest Set` определяется по максимальному весу, затем по повторениям;
- `Recent Volume Trend` содержит максимум 7 последних тренировок диапазона.

### 13.4. Инварианты по вкладке New

- в одной тренировке одно упражнение нельзя добавить дважды как отдельную сущность;
- `Стандарт` доступен только при наличии валидной истории по упражнению;
- `Стандарт` всегда добавляет подход `12 reps`;
- стартовый вес модального окна берется из последней подходящей тренировки;
- `set_index` в новой тренировке всегда последовательный и начинается с 1;
- после сохранения тренировка сразу участвует в `Trainings`, `Progress` и в логике `Стандарт`.

## 14. Ограничения текущей версии

Текущая локальная версия не поддерживает:

- удаление тренировок;
- редактирование уже сохраненной тренировки;
- редактирование или удаление отдельного подхода;
- редактирование `notes`, `focus`, `plan_id`;
- серверную синхронизацию;
- многопользовательский режим;
- отдельный бизнес-сценарий для `deload`.

## 15. Что считать источником истины

Если в будущем поведение документа и приложения разойдется, для текущей web-версии источником истины следует считать:

- сценарии пользователя, описанные в этом документе;
- текущее поведение `telegram_miniapp/web/main.js`;
- контракт данных фикстур и схемы тренировки.

Для изменений бизнес-логики желательно сначала обновлять этот документ, а затем реализацию и тесты.
