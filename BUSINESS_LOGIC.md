# Trainer Business Logic

## 1. Зачем нужен этот документ

Этот документ фиксирует текущее продуктовое поведение `Trainer`:

- как приложение авторизует пользователя;
- какие данные считаются источником истины;
- как устроены экраны `Trainings`, `Progress`, `Weight` и flow создания тренировки;
- какие инварианты должны быть зафиксированы в тестах;
- что сейчас считается частью продукта, а что сохранено в коде, но отключено в UI.

Технические детали окружения, запуска и деплоя лежат в:

- [backend/README.md](./backend/README.md)

## 2. Общая модель продукта

`Trainer` — это backend API на SQLite + нативный iOS-клиент.

В текущей версии:

- справочник упражнений загружается из JSON;
- сохранённые тренировки и вес тела хранятся на backend;
- черновик новой тренировки и часть UI-состояний живут локально на клиенте;
- одна база используется для нескольких пользователей, но данные изолированы по `user_id`;
- основной режим — нативный iOS-клиент (personal-build, фиксированный пользователь);
- браузерный debug-режим нужен для локальной разработки и отладки.

## 3. Режимы пользователя и сессии

### 3.1. Native iOS fixed user

Основной режим. Нативный iOS-клиент работает без внешней авторизации:

- iOS отправляет `shell=ios` и `native_user_id=3` в `POST /api/session/resolve`;
- backend находит пользователя с таким `id`;
- backend выдаёт подписанную cookie `trainer_session`;
- дальнейшие workout/body-weight endpoints работают в контексте этого `user_id`.

Это сознательный personal-build режим. Он не предназначен для публичного multi-user доступа.

### 3.2. Browser debug user

Для локальной разработки браузер без сессии может автоматически получать debug-сессию
(при `MINIAPP_ALLOW_DEBUG_USER=1` или `MINIAPP_DEV_MODE=1`):

- по умолчанию используется alias `browser-default`, если не задан другой;
- данные debug-user живут отдельно.

## 4. Основные сущности

### 4.1. Упражнение

Упражнение из справочника содержит:

- `id`
- `name`

### 4.2. Подход

Подход содержит:

- `set_index`
- `weight`
- `reps`
- `effort`
- `notes`

`effort` — это нормализованная дискретная оценка тяжести:

- `easy`
- `ok`
- `hard`
- `null`

На UI это отображается как:

- `🙂`
- `😐`
- `😣`

### 4.3. Тренировка

Тренировка содержит:

- `id`
- `client_id`
- `workout_date`
- `created_at`
- `updated_at`
- `data.load_type`
- `data.exercises`

### 4.4. Запись веса тела

Запись веса тела содержит:

- `id`
- `entry_date`
- `weight`
- `created_at`
- `updated_at`

Для одного пользователя на одну дату существует только одна актуальная запись. Повторное сохранение на ту же дату обновляет значение, а не создаёт дубль.

## 5. Источники данных

### 5.1. Справочник упражнений

Источник:

- [backend/static/data/exercises.json](./backend/static/data/exercises.json)

Используется для:

- основной подборки и редкого каталога упражнений;
- preview- и draft-карточек в `New Workout`;
- селектора упражнения на экране `Progress`.

### 5.2. Серверные тренировки

Источник истины для:

- истории тренировок;
- экрана `Progress`;
- reference-строк `прошлое выполнение → план`;
- быстрого planned add;
- кольцевого прогресса вокруг FAB.

Источник:

- SQLite backend;
- таблица тренировок текущего пользователя.

### 5.3. Серверные записи веса тела

Источник истины для экрана `Weight`:

- SQLite backend;
- таблица `body_weights`.

### 5.4. Что хранится локально

В `localStorage` живут только UI-состояния:

- черновик тренировки;
- текущая вкладка;
- выбранный диапазон на `Progress`;
- выбранный диапазон на `Weight`;
- выбранное упражнение на `Progress`.

Сохранённые тренировки и записи веса тела не считаются локальным источником истины.

## 6. Навигационная модель

Нижний бар состоит из трёх вкладок:

- `Trainings`
- `Progress`
- `Weight`

Создание новой тренировки не является отдельной вкладкой.

Новая тренировка открывается:

- по FAB `+` на `Trainings`;
- либо через возврат в уже существующий черновик.

По умолчанию приложение открывается на `Trainings`.

## 7. Правила сортировки данных

### 7.1. Сортировка тренировок

Тренировки сортируются:

1. по `workout_date` по убыванию;
2. при совпадении даты — по `created_at` по убыванию;
3. затем по `updated_at` по убыванию;
4. затем по `id` по убыванию.

Это правило используется в:

- `Trainings`;
- логике поиска последнего выполнения упражнения;
- аналитике прогресса.

### 7.2. Сортировка записей веса тела

Записи веса тела для графика упорядочиваются от старых к новым.

## 8. Экран Trainings

### 8.1. Назначение

Экран показывает историю тренировок текущего пользователя.

### 8.2. Состав карточки тренировки

Карточка включает:

- дату и день недели в одной строке;
- компактный список упражнений;
- сгруппированное summary по сетам.

`load_type` по-прежнему вычисляется и хранится в данных тренировки, но текущий UX истории не показывает отдельный `heavy / medium / light` badge.

### 8.3. Формат summary по сетам

Подходы группируются по последовательным одинаковым весам. Внутри такой группы повторения схлопываются в компактную строку.

Примеры:

- `60кг ×14`
- `60кг ×14×3`
- `90кг ×13×2, 12`

Если у подходов есть `effort`, он приклеивается к соответствующей компактной группе.

Если у подходов есть `notes`, они выводятся отдельной компактной строкой под упражнением.

### 8.4. Swipe actions

Карточка тренировки поддерживает свайп влево и открывает два действия:

- редактирование;
- удаление.

Карточка должна уметь закрываться назад:

- свайпом вправо;
- тапом по свободной области.

### 8.5. FAB на экране истории

Внизу справа расположен FAB:

- если черновика с реальными сетами нет, это обычный `+`;
- если живой черновик есть, FAB становится входом обратно в него.

Если есть живой черновик, вокруг FAB показывается кольцевой прогресс по основной шестёрке упражнений.

Этот прогресс считается дробно:

- по упражнениям основной шестёрки;
- по количеству выполненных planned sets внутри каждого упражнения;
- а не просто по факту “в упражнении появился хотя бы один сет”.

### 8.6. Next workout plan

В коде существует генератор `next workout plan`, но текущий UI его не показывает.

Фича отключена флагом:

- `ENABLE_NEXT_WORKOUT_PLAN = false`

### 8.7. Совет тренера (LLM-рекомендация)

Отдельная фича поверх эвристик: backend строит план **следующей** тренировки по истории
пользователя через Claude API и кэширует его (одна актуальная рекомендация на пользователя).

- **Состав рекомендации** повторяет тренировку: `exercises[]` с `exercise_id`/`name`/`note`
  и `sets[{reps, weight}]`, плюс `focus`, `load_type` (heavy/medium/light) и развёрнутый
  `rationale` — текстовое объяснение, почему выбран именно такой состав и нагрузка, а не другой.
- **Только каталожные упражнения.** `exercise_id` ограничен enum'ом из 12 упражнений на уровне
  JSON-схемы; на сервере результат дополнительно валидируется (выдуманные id отбрасываются,
  reps/weight приводятся к разумным границам, имя берётся из каталога).
- **Свежесть.** Рекомендация помечается `stale`, если появилась тренировка новее той, по которой
  она построена. После создания/изменения/удаления тренировки она перегенерируется в фоне.
- **iOS UX.** Карточка «Совет тренера» вверху экрана `Trainings`: читается мгновенно из кэша
  (вне 3-сек дедлайна загрузки), «Обновить» запускает синхронную генерацию (10–40 с, оверлей).
  Состояния: `none` / `pending` / `ready` (+ `stale`) / `failed`.
- **Применение как план.** «Применить в план» делает совет планом (`appliedPlan`), НЕ
  выполненными подходами: тренировка не считается начатой, пока не записан первый сет.
  Карточки идут в порядке рекомендации с целями «вес × повторы», заголовок секции —
  «План от тренера» (+ «Сбросить»; превью-карточку можно убрать из плана long press-ом).
  Кнопка совета в применённом состоянии — «В плане ✓», для изменившегося совета —
  «Применить новый» (заменяются только цели, записанные сеты не трогаются).
- **Следующий подход по «+»** (единый приоритет и для конструктора): последний кастомный
  подход повторяется → цель применённого плана по индексу → история +1 повтор → 12×0.
  Long press по «+» открывает конструктор подхода.
- **Связка с тренировкой.** При сохранении новой тренировки с применённым планом в payload
  пишется `data.recommendation` — снапшот совета (без rationale/notes) для статистики
  «факт vs план»; сервер сохраняет снапшот при последующих PUT-редактированиях. После
  сохранения план очищается, рекомендация перегенерируется в фоне.
- **Источник данных для модели** — последние N тренировок (по умолчанию 20), сериализованные
  компактно с пометками тяжести (`-`/`+`), плюс динамика веса тела.

## 9. Конструктор новой тренировки

### 9.1. Вход в flow

Экран `New Workout` открывается из `Trainings` по FAB.

Если в `localStorage` уже есть живой черновик, пользователь попадает в него, а не в пустой flow.

### 9.2. Дата тренировки

Дата редактируется из компактного чипа в верхнем баре.

Отдельной большой карточки даты на экране нет.

### 9.3. Стартовое состояние

Если пользователь открывает `New Workout` без уже внесённых сетов, экран не должен быть пустым.

Вместо пустого состояния конструктор сразу показывает preview-карточки основных упражнений, как будто пользователь уже выбрал их из основной подборки, но ещё не добавил ни одного сета.

Редкие упражнения доступны отдельно через блок `Ещё упражнения`.

### 9.4. Как формируется основная подборка

Основная шестёрка строится по реальной истории тренировок пользователя:

- приоритет получают часто встречающиеся упражнения;
- дополнительно учитывается их типичная позиция внутри тренировки;
- целевой размер основной подборки — до 6 упражнений;
- `Жим гор.` всегда остаётся в редких упражнениях.

### 9.5. Preview-карточки и редкий каталог

- упражнение не должно дублироваться в текущем черновике;
- preview-карточка показывает подсказки и кнопку planned add, но не считается начатым упражнением, пока не появился реальный сет;
- после первого сохранённого сета preview-карточка превращается в обычную draft-карточку;
- если пользователь открыл set-modal, но закрыл его без сохранения первого сета, пустая карточка не должна остаться в черновике;
- `Ещё упражнения` открывает отдельный редкий каталог и не должен пополнять основную шестёрку;
- когда все упражнения основной шестёрки реально начаты, показывается completion-state `Круто, тренировка закончена`, но редкий каталог остаётся доступным.

### 9.6. Суперсеты

Конструктор поддерживает параллельное ведение нескольких упражнений:

- не нужно “заканчивать” одно упражнение, чтобы начать второе;
- пользователь может чередовать сеты между карточками;
- у каждой карточки своё текущее состояние, собственный planned add и своя история progress/reference.

### 9.7. Единый источник плановой логики

План по упражнению должен считаться в одном месте и переиспользоваться везде одинаково.

Один и тот же helper обязан обслуживать:

- reference-строку в карточке упражнения;
- reference-строку в set-modal;
- быстрый planned add по синему `+`;
- стартовые значения ручного set-modal;
- кольцевой прогресс вокруг FAB;
- потенциальный возврат UI `next workout plan`, если фича снова будет включена.

### 9.8. Карточка упражнения в конструкторе

Карточка упражнения содержит:

- заголовок с названием и маленькой пиктограммой упражнения;
- reference-строку `прошлое выполнение → план`;
- компактную синюю строку уже внесённых сетов;
- один большой синий `+` справа.

Summary уже внесённых сетов использует ту же логику схлопывания, что и история тренировок.

Если синяя строка умещается в одну линию, она должна оставаться одной строкой. Если не умещается, она может аккуратно перейти на две строки.

### 9.9. Формат reference-строки

Reference-строка строится из двух частей:

- серая часть — прошлое выполнение;
- зелёная часть — planned target на текущую тренировку.

Важно:

- смайлики `effort` относятся только к прошлому выполнению;
- в плановую зелёную часть они не копируются;
- строка должна стараться оставаться в одну линию;
- если не помещается, допускается перенос на две строки без визуального развала по вертикали.

### 9.10. Быстрое добавление planned set

Короткий тап по синему `+` не открывает модалку, а сразу добавляет следующий planned set.

Правила:

- источник плана — последнее подходящее выполнение упражнения;
- очередность идёт по порядку прошлых сетов;
- в каждый planned set добавляется `+1` к повторам;
- `effort` из прошлого выполнения не копируется в planned set;
- если в истории по упражнению ничего нет, fallback остаётся простым: используется последний известный вес и `12` повторений.

### 9.11. Ручной ввод сета по long press

Long press по синему `+` открывает ручной set-modal.

Ручной ввод не должен открываться по случайному тапу в любую область карточки. Только по long press на самой кнопке `+`.

### 9.12. Set-modal

Актуальная модалка сета устроена так:

- закрывается тапом вне модалки;
- отдельной кнопки `Отмена` нет;
- отдельного заголовка `Новый сет` / `Редактировать сет` нет;
- сверху показывается название упражнения с иконкой и reference-строка;
- ниже идут две компактные stepper-строки:
  - вес в формате `120,5 кг`;
  - повторы в формате `11 ПТ`;
- stepper-кнопки используют тот же общий синий круглый стиль, что и `+` на карточке упражнения;
- внутри модалки можно выбрать `effort` через `🙂 / 😐 / 😣`;
- основное действие модалки только одно: `Применить`.

### 9.13. Long press по карточке упражнения

Long press по поверхности карточки упражнения открывает центрированную modal-карточку в том же визуальном паттерне, что и set-modal.

В ней доступны два действия:

- удалить последний сет;
- удалить всё упражнение.

Отдельный draft-swipe для удаления больше не является частью актуального UX.

### 9.14. FAB-ы внутри конструктора

Пока в черновике нет ни одного реального сета, FAB-ы не показываются.

После появления первого сета становятся видны:

- чёрный save-FAB справа;
- красный reset-FAB слева.

### 9.15. Reset-flow

Красная кнопка не сбрасывает черновик мгновенно.

Она сначала показывает confirm, и только после подтверждения:

- очищает черновик;
- возвращает экран в исходное состояние.

### 9.16. Кольцевой прогресс вокруг FAB

Save-FAB и FAB на `Trainings` могут быть окружены кольцевым прогрессом.

Правила:

- прогресс считается только по упражнениям основной шестёрки;
- вклад упражнения зависит от доли выполненных planned sets относительно плана этого упражнения;
- простое открытие модалки без применения сета прогресс не увеличивает;
- быстрое planned add и ручное сохранение сета увеличивают прогресс одинаково через общую плановую логику;
- при удалении сетов или упражнения прогресс должен убывать симметрично;
- кольцо анимируется вперёд и назад при изменении состояния черновика.

### 9.17. Автоскролл конструктора

Если пользователь добавляет новое упражнение и его карточка оказывается вне видимой области, экран должен подскроллиться так, чтобы новая карточка оставалась выше нижних FAB-ов.

### 9.18. Сохранение тренировки

При сохранении:

- frontend отправляет тренировку на backend;
- backend нормализует payload;
- тренировка получает серверный `id`;
- черновик очищается;
- новая запись сразу появляется в `Trainings`, `Progress` и влияет на будущий planned add.

Save-FAB должен показывать явное loading-состояние, а не выглядеть как зависший UI.

### 9.19. Автоматический `load_type`

Если `load_type` не задан явно, backend выводит его из суммарного объёма:

- `heavy` — если объём `>= 3000`;
- `medium` — если объём `>= 1600` и `< 3000`;
- `light` — если объём `< 1600`.

В текущем UX это серверное поле сохраняется, но не обязано быть визуально видимым на истории тренировок.

## 10. Экран Progress

### 10.1. Назначение

Экран отвечает на два вопроса:

- сколько тренировок было в выбранном диапазоне;
- как растут рабочий вес и повторения по конкретному упражнению.

### 10.2. Диапазоны

Доступны:

- `7D`
- `30D`
- `All`

### 10.3. Summary

В summary сейчас показывается только:

- количество тренировок за период.

Старые объёмные метрики вроде `Volume`, `Top Exercise`, `Heaviest Set` и `Average Volume` не считаются частью актуального продукта.

### 10.4. Выбор упражнения

Селектор строится из реальной серверной истории пользователя.

Каталог упражнений может использоваться как fallback, но основной смысл экрана — анализ уже реально встречавшихся упражнений.

### 10.5. Серии на графике

Для каждой тренировки в выбранном диапазоне строятся две линии:

- `Вес` — самый тяжёлый сет упражнения на тренировке;
- `Повторы` — сет с максимумом повторений на тренировке.

### 10.6. Snapshot-карточки

Под выбранное упражнение показываются три snapshot-карточки:

- `Старт`
- `Сейчас`
- `Изменение`

Они отражают первый и последний points текущего диапазона.

## 11. Экран Weight

### 11.1. Назначение

Экран отвечает за контроль веса тела.

### 11.2. Основные элементы

Экран содержит:

- диапазоны `7D / 30D / All`;
- компактный inline composer;
- summary-strip;
- график веса тела.

### 11.3. Inline composer

Composer состоит из:

- чипа с датой;
- поля ввода веса;
- кнопки сохранения.

Надзаголовков и лишних поясняющих текстов в актуальном UI нет.

### 11.4. Правило сохранения веса

На одну дату у пользователя существует только одна запись.

Поэтому повторное сохранение на ту же дату:

- не создаёт дубль;
- обновляет уже существующее значение.

### 11.5. Summary-strip

На экране показываются три компактные метрики:

- `Последний`
- `Дельта`
- `Записей`

### 11.6. График веса тела

График показывает:

- только реальные записи пользователя;
- по оси `Y` — только реально введённые значения веса, без искусственных промежуточных меток;
- по оси `X` — даты текущего диапазона данных.

### 11.7. Удаление записи веса

Тап по точке графика открывает confirm на удаление записи.

При подтверждении:

- запись удаляется на backend;
- график и summary сразу пересчитываются.

## 12. Правила отображения чисел

### 12.1. Рабочие веса в тренировках

Вес отображается компактно, без пробела перед `кг`.

Примеры:

- `60кг ×14`
- `60кг ×14×3`

### 12.2. Повторы

Повторы отображаются целыми числами.

### 12.3. Вес тела

Для веса тела UI не должен навязывать искусственное округление осей до одной цифры после запятой, если пользователь вводил более точное значение.

## 13. Загрузка и ошибки

### 13.1. Старт приложения

На старте приложение:

1. инициализирует Telegram shell, если он доступен;
2. пытается вызвать `disableVerticalSwipes()` и `requestFullscreen()`, если клиент это поддерживает;
3. восстанавливает черновик и UI-состояние;
4. резолвит пользовательскую сессию;
5. загружает справочник упражнений, тренировки и вес тела;
6. строит итоговое состояние.

### 13.2. Loading-screen

Во время старта показывается отдельный branded loading-screen.

### 13.3. Ошибка загрузки

Если boot не собрался, показывается экран ошибки с возможностью повторить загрузку.

## 14. Инварианты для тестов

### 14.1. Пользовательские данные

- данные одного пользователя не должны попадать в контекст другого;
- Telegram-shell не должен “залипать” на debug-user;
- recovery и fallback должны быть предсказуемыми и покрыты тестами.

### 14.2. Trainings

- список всегда берётся с backend;
- карточки отсортированы по правилу свежести;
- swipe edit/delete работают;
- summary сетов компактно группируется;
- новая тренировка сразу появляется в истории;
- load-badge не должен случайно вернуться в текущий UX, пока продуктово он выключен.

### 14.3. New Workout

- FAB-ы не появляются до первого реального сета;
- стартовый экран конструктора должен сразу показывать preview-карточки основной шестёрки, а не пустую плитку;
- пустое выбранное упражнение не должно сохраняться после закрытия модалки без применения первого сета;
- короткий тап по синему `+` должен добавлять следующий planned set, а long press по нему открывать ручной modal-flow;
- выбранный `effort` должен сохраняться в set и отображаться в текущем draft и в истории тренировок;
- summary введённых сетов в draft-карточке должно использовать ту же логику схлопывания, что и история тренировок;
- tap по синей summary-строке должен редактировать именно последний сет упражнения;
- long press по карточке упражнения должен открывать центрированную modal-карточку удаления;
- completion-state основной подборки должен появляться только после реального исчерпания упражнений основной шестёрки;
- автоскролл должен удерживать новые карточки выше FAB-ов;
- прогресс-кольцо должно расти и убывать пропорционально выполненным planned sets, а не просто по наличию любого сета;
- при возвращении к живому черновику пользователь попадает в него, а не в новый пустой flow.

### 14.4. Progress

- число тренировок в summary равно числу тренировок в выбранном диапазоне;
- для выбранного упражнения линии строятся по лучшему весу и лучшим повторам на тренировку;
- если данных нет, экран должен показывать понятное пустое состояние.

### 14.5. Weight

- запись на ту же дату должна обновляться, а не дублироваться;
- график должен строиться по реальным данным пользователя;
- удаление по тапу на точку должно реально удалять запись на backend.

## 15. Текущие ограничения

- эвристический `Плана следующей тренировки` отключён в UI (код сохранён); вместо него работает LLM-«Совет тренера» (см. §8.7);
- авторизации как таковой нет: iOS работает в personal-build режиме с фиксированным `native_user_id`, браузер — в debug-режиме;
- одна запись веса тела на дату — это текущая продуктовая модель, а не история нескольких измерений за день;
- `focus` и комментарии уровня всей тренировки пока не вводятся пользователем в UI (AI-«Совет тренера» уже есть, см. §8.7);
- `effort` сейчас задаётся только через три дискретных состояния, без свободного текстового комментария к сету.
