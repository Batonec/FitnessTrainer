# Trainer Business Logic

## 1. Назначение документа

Этот документ фиксирует текущую бизнес-логику приложения `Trainer`.

Документ нужен как опорная спецификация для:

- понимания пользовательского поведения;
- генерации тестов;
- проверки регрессий;
- синхронизации frontend, backend и Telegram Mini App-режима по смыслу, а не по деталям реализации.

Документ описывает наблюдаемое поведение системы и правила обработки данных. Он не заменяет техническую документацию по деплою и инфраструктуре.

## 2. Текущий режим работы

Приложение работает как Telegram Mini App с backend API на SQLite.

На текущем этапе:

- справочник упражнений и стартовая история тренировок загружаются из JSON-фикстур;
- пользовательские тренировки сохраняются на backend;
- приложение может работать как в Telegram, так и в обычном браузере для локальной разработки;
- часть UI-состояния хранится локально в браузере;
- одна база данных используется для всех пользователей, но данные изолированы по `user_id`.

## 3. Режимы пользователя

### 3.1. Telegram user

Основной целевой режим:

- Mini App открыт из Telegram;
- frontend получает `Telegram.WebApp.initData`;
- backend валидирует `initData`;
- пользователь связывается с записью в таблице `users`;
- все операции с тренировками выполняются в контексте этого Telegram-пользователя.

### 3.2. Telegram fallback user

Если Mini App открылся в Telegram, но signed `initData` недоступен, приложение может использовать fallback-режим:

- frontend отправляет `unsafeUser` из Telegram SDK;
- backend создает или обновляет пользователя с `auth_source = telegram_unsafe`;
- пользователь получает отдельную серверную сессию;
- данные такого пользователя тоже изолированы от других пользователей.

Этот режим считается временной совместимой деградацией и не является предпочтительным режимом продовой авторизации.

### 3.3. Default browser user

Для локальной отладки браузер без Telegram может работать как дефолтный debug-пользователь:

- backend создает или переиспользует пользователя с `debug_alias = browser-default`;
- frontend считает этого пользователя текущим активным пользователем;
- все тренировки в браузерном debug-режиме сохраняются в отдельный серверный набор данных этого пользователя.

## 4. Основные сущности

### 4.1. Упражнение

Упражнение в справочнике содержит:

- `id` — числовой идентификатор упражнения;
- `name` — отображаемое название упражнения.

### 4.2. Подход

Подход содержит:

- `set_index` — порядковый номер подхода внутри упражнения;
- `reps` — количество повторений;
- `weight` — рабочий вес;
- `notes` — опциональная заметка.

### 4.3. Упражнение внутри тренировки

Упражнение внутри тренировки содержит:

- `exercise_id`;
- `name`;
- список подходов `sets`.

### 4.4. Тренировка

Тренировка содержит:

- `id` — серверный идентификатор тренировки;
- `workout_date` — дата тренировки в формате `YYYY-MM-DD`;
- `plan_id` — сейчас всегда `null`;
- `created_at` — время создания на сервере;
- `updated_at` — время последнего обновления на сервере;
- `data.focus` — сейчас всегда `null`;
- `data.notes` — сейчас всегда `null`;
- `data.load_type` — тип нагрузки;
- `data.exercises` — список упражнений с подходами.

### 4.5. Тип нагрузки

Допустимы значения:

- `heavy`
- `medium`
- `light`
- `deload`

В текущем UI:

- активно используются `heavy`, `medium`, `light`;
- `deload` допустим по контракту, но не имеет отдельной особой визуальной логики;
- если значение отсутствует, бейдж нагрузки не показывается.

## 5. Источники данных

### 5.1. Фикстуры

Статические JSON-файлы:

- `telegram_miniapp/web/data/exercises.json`
- `telegram_miniapp/web/data/workouts.json`

Они задают:

- справочник упражнений;
- стартовую историю тренировок, одинаковую для всех пользователей.

### 5.2. Серверные пользовательские тренировки

Все новые тренировки сохраняются на backend в SQLite.

Они:

- принадлежат конкретному пользователю;
- не заменяют фикстуры;
- добавляются к ним в итоговом наборе данных.

### 5.3. Объединение данных

Во всех экранах приложение работает с объединенным списком тренировок:

- сначала берутся пользовательские тренировки текущего пользователя с backend;
- затем добавляются тренировочные фикстуры;
- итоговый список сортируется единым правилом.

## 6. Правило сортировки тренировок

Тренировки сортируются так:

1. по `workout_date` по убыванию;
2. если дата совпадает, по `created_at` по убыванию;
3. если `created_at` совпадает или отсутствует, по `updated_at` по убыванию;
4. если timestamps отсутствуют у одной из записей, серверные тренировки считаются новее фикстур;
5. если и этого недостаточно, по `id` по убыванию.

Это правило применяется для:

- списка `Trainings`;
- выбора последнего рабочего веса;
- расчета прогресса;
- порядка новых тренировок внутри одного дня.

## 7. Локальное хранение состояния

### 7.1. Что хранится локально

В `localStorage` хранятся только элементы UI-состояния:

- `trainer-miniapp-draft-v1` — черновик незавершенной тренировки;
- `trainer-miniapp-tab-v1` — последняя открытая вкладка;
- `trainer-miniapp-range-v1` — выбранный диапазон на вкладке `Progress`.

### 7.2. Что больше не хранится локально

Пользовательские тренировки больше не являются источником истины в `localStorage`.

Если раньше такие записи были сохранены локально, при старте они переносятся на backend и затем удаляются из локального старого ключа.

### 7.3. Бизнес-смысл локального состояния

Это означает:

- пользователь может перезагрузить страницу и не потерять черновик;
- приложение помнит вкладку и диапазон прогресса;
- черновик восстанавливается как продолжение незавершенной тренировки, а не как отдельная сохраненная тренировка.

## 8. Структура приложения

Приложение состоит из трех основных вкладок:

- `Trainings`
- `Progress`
- `New`

По умолчанию при первом запуске открывается вкладка `New`.

## 9. Вкладка Trainings

### 9.1. Назначение

Вкладка показывает объединенную историю тренировок:

- фикстуры;
- серверные тренировки текущего пользователя.

### 9.2. Что отображается в карточке тренировки

Каждая карточка содержит:

- полную дату;
- день недели;
- бейдж нагрузки, если `load_type` заполнен;
- список упражнений;
- внутри упражнения — список сгруппированных подходов.

### 9.3. Формат даты

Дата отображается в длинном русском формате.

Пример:

- `28 марта 2026`

День недели отображается отдельной строкой и начинается с заглавной буквы.

Пример:

- `Суббота`

### 9.4. Отображение бейджа нагрузки

Если у тренировки есть `load_type`, он показывается как верхний бейдж:

- `LIGHT`
- `MEDIUM`
- `HEAVY`
- либо другое пришедшее значение в верхнем регистре

### 9.5. Группировка подходов в истории

Подходы внутри упражнения объединяются только если идут подряд и совпадают по:

- `weight`;
- `reps`;
- `notes`.

Если между одинаковыми подходами встречается другой подход, создаются отдельные группы.

### 9.6. Подпись сгруппированного подхода

Если в группе один подход:

- `Вес кг × Повторения`

Если в группе несколько подряд одинаковых подходов:

- `Вес кг × Повторения × Количество`

Примеры:

- `80 кг × 8`
- `80 кг × 8 × 3`

### 9.7. Заметки к подходу

Если у группы есть `notes`, заметка отображается справа от строки подхода.

### 9.8. Пустое состояние

Если тренировок нет вообще, показывается пустое состояние с сообщением, что первая тренировка появится после сохранения.

## 10. Вкладка Progress

### 10.1. Назначение

Вкладка показывает агрегированную статистику по объединенной истории тренировок.

### 10.2. Диапазоны анализа

Доступны три диапазона:

- `7D`
- `30D`
- `All`

### 10.3. Фильтр по диапазону

Для `7D` и `30D` используется включительный интервал:

- верхняя граница — сегодняшняя локальная дата;
- нижняя граница — `today - (days - 1)`.

Это значит:

- `7D` включает сегодня и ещё 6 предыдущих дней;
- `30D` включает сегодня и ещё 29 предыдущих дней;
- `All` включает все тренировки.

### 10.4. Метрики

Рассчитываются:

- `Workouts` — число тренировок в диапазоне;
- `Volume` — суммарный объём;
- `Average Volume / Workout` — средний объём на тренировку;
- `Top Exercise` — упражнение с максимальным суммарным объёмом;
- `Heaviest Set` — самый тяжёлый подход;
- `Recent Volume Trend` — объём по максимум 7 последним тренировкам диапазона.

### 10.5. Что считается объёмом

Объём одного подхода:

- `weight * reps`

Подход участвует в объёме только если:

- `weight > 0`
- `reps > 0`

Объём тренировки:

- сумма валидных подходов всех упражнений тренировки.

### 10.6. Top Exercise

Для каждого упражнения с одинаковым `name` суммируется объём в диапазоне.

Побеждает упражнение с максимальным суммарным объёмом.

Если тренировок нет, `Top Exercise` не показывается.

### 10.7. Heaviest Set

Самым тяжёлым считается подход:

1. с максимальным `weight`;
2. при равном `weight` — с максимальным `reps`.

Если тренировок нет, `Heaviest Set` не показывается.

### 10.8. Recent Volume Trend

Для тренда:

- тренировки диапазона сортируются от новых к старым;
- берутся максимум 7 последних;
- затем они разворачиваются в порядок от более старой к более новой внутри этого набора.

### 10.9. Поведение при отсутствии данных

Если в диапазоне нет тренировок:

- числовые метрики становятся нулевыми;
- `Top Exercise` и `Heaviest Set` не отображаются;
- вместо тренда показывается `No workouts in this range`.

### 10.10. Ручное обновление

Кнопка обновления:

- заново резолвит текущую пользовательскую сессию;
- перечитывает фикстуры;
- перечитывает серверные тренировки;
- не удаляет черновик тренировки.

## 11. Вкладка New

### 11.1. Назначение

Вкладка предназначена для сборки новой тренировки.

### 11.2. Базовый сценарий

Пользователь:

1. выбирает упражнение;
2. добавляет подходы;
3. при необходимости переходит к следующему упражнению;
4. завершает тренировку;
5. получает серверно сохраненную тренировку, которая сразу участвует в истории и прогрессе.

### 11.3. Автоматическое открытие выбора упражнения

Если:

- справочник упражнений уже загружен;
- черновик пуст;
- активное упражнение ещё не выбрано;

интерфейс автоматически открывает picker выбора упражнения.

### 11.4. Ограничение на повторное добавление упражнения

В рамках одной тренировки одно и то же упражнение нельзя добавить как отдельную сущность дважды.

Поэтому picker всегда исключает упражнения, уже входящие в текущий черновик.

### 11.5. Состояние выбранного упражнения

После выбора упражнения оно становится активным.

Если у него ещё нет подходов, отображается карточка:

- название упражнения;
- подпись `Сетов пока нет`.

### 11.6. Доступные действия для активного упражнения

Если упражнение выбрано и интерфейс не находится:

- в picker-е выбора упражнения;
- в модальном окне добавления подхода;

то доступны:

- `Добавить сет`
- `Стандарт` — только если есть валидная история

### 11.7. Что считается валидной историей для Стандарта

Кнопка `Стандарт` доступна, если в истории есть хотя бы один подход этого упражнения с условиями:

- `weight > 0`
- `reps > 0`

### 11.8. Как выбирается стартовый вес

Берётся история тренировок, отсортированная по общему правилу от новых к старым.

Для выбранного упражнения находится первая подходящая тренировка.

Из неё берётся:

- максимальный `weight` среди всех подходов этого упражнения.

Если подходящего веса нет:

- стартовый вес равен `0`.

### 11.9. Логика окна добавления подхода

При открытии модального окна:

- `reps` по умолчанию равен `12`;
- `weight` по умолчанию равен последнему подходящему рабочему весу.

Пользователь может менять:

- вес шагом `2.5`;
- повторения шагом `1`.

Ограничения:

- вес не может быть меньше `0`;
- повторения не могут быть меньше `1`.

### 11.10. Логика кнопки Стандарт

Кнопка `Стандарт` сразу добавляет подход без модального окна.

Параметры:

- `reps = 12`
- `weight = weightFromLastWorkout`
- `notes = null`

### 11.11. Добавление подхода в упражнение

Если упражнение ещё не существует в черновике:

- создаётся новая запись упражнения.

Если уже существует:

- новый подход дописывается в конец.

### 11.12. Группировка подходов на экране New

На вкладке `New` используется отдельная группировка:

- учитываются только `weight` и `reps`;
- `notes` в группировке не участвуют;
- подходы объединяются только если идут подряд.

Формат:

- один подход: `Вес кг × Повторения`
- несколько одинаковых подряд: `Вес кг × Повторения × Количество`

### 11.13. Переход к следующему упражнению

После того как у текущего упражнения появился хотя бы один подход, становится доступна кнопка `Новое упражнение`.

Она:

- сохраняет уже собранные данные;
- переводит интерфейс в picker следующего упражнения;
- показывает только упражнения, которых ещё нет в черновике.

### 11.14. Восстановление черновика

Если пользователь перезагрузил страницу или заново открыл приложение, незавершённый черновик восстанавливается.

В этом случае:

- показывается карточка `Восстановлен черновик тренировки`;
- пользователь видит, сколько упражнений уже добавлено;
- picker объясняет, что ниже показаны только оставшиеся упражнения.

### 11.15. Сброс черновика

Пользователь может нажать `Начать заново`.

Это действие:

- удаляет текущий черновик из `localStorage`;
- очищает активное упражнение и уже добавленные упражнения;
- возвращает интерфейс к чистому состоянию новой тренировки;
- снова показывает полный список упражнений.

### 11.16. Состояние “все упражнения уже добавлены”

Если в черновик уже включены все упражнения справочника:

- picker новых упражнений больше не предлагает новых вариантов;
- пользователь может только завершить тренировку или начать заново.

### 11.17. Завершение пустой тренировки

Если пользователь нажимает `Закончить тренировку`, но упражнений нет:

- тренировка не создаётся;
- черновик очищается;
- интерфейс возвращается в исходное состояние.

### 11.18. Завершение непустой тренировки

Если в черновике есть упражнения:

- frontend отправляет тренировку на backend;
- backend нормализует данные;
- тренировке назначается серверный `id`;
- черновик очищается;
- показывается flash-сообщение об успешном сохранении.

### 11.19. Правила сохранения новой тренировки

При отправке формируется объект со следующими правилами:

- `client_id` уникален на клиенте;
- `workout_date` равен текущей локальной дате;
- `plan_id = null`;
- `focus = null`;
- `notes = null`;
- `sets[].set_index` начинается с `1` и идёт последовательно.

### 11.20. Автоматическое определение load_type

Если `load_type` не задан явно, backend определяет его по суммарному объёму:

- `heavy`, если объём `>= 3000`
- `medium`, если объём `>= 1600` и `< 3000`
- `light`, если объём `< 1600`

## 12. Поведение после сохранения тренировки

После серверного сохранения новая тренировка должна:

- появиться в `Trainings`;
- участвовать в расчётах `Progress`;
- учитываться в логике `Стандарт`;
- учитываться при вычислении стартового веса следующего подхода.

Это ключевая бизнес-инварианта.

## 13. Правила отображения чисел

### 13.1. Вес

Вес отображается:

- без дробной части, если число целое;
- с одной цифрой после запятой, если есть половинный шаг.

Примеры:

- `80 кг`
- `82,5 кг`

### 13.2. Повторения

Повторения отображаются целым числом.

### 13.3. Объём

Объём отображается как целая часть значения:

- дробная часть отбрасывается;
- затем добавляется суффикс `kg`.

Пример:

- `1547 kg`

## 14. Состояния загрузки и ошибки

### 14.1. Стартовая загрузка

На старте приложение:

1. поднимает Telegram shell, если он доступен;
2. восстанавливает черновик;
3. резолвит пользовательскую сессию;
4. переносит старые локальные тренировки на backend, если они ещё остались;
5. загружает фикстуры и серверные тренировки;
6. строит итоговое состояние.

### 14.2. Ошибка загрузки

Если не удалось собрать приложение:

- показывается экран ошибки;
- на экране есть кнопка `Повторить`.

### 14.3. Повтор после ошибки

`Повторить` должен:

- заново попытаться резолвить сессию;
- заново перечитать данные;
- восстановить рабочее состояние, если проблема исчезла.

## 15. Бизнес-инварианты для тестов

### 15.1. Инварианты по данным

- пользовательские тренировки не заменяют фикстуры, а добавляются к ним;
- данные одного пользователя не должны попадать другому пользователю;
- одинаковый `client_id` у одного пользователя не должен создавать дубликат;
- объединённый список тренировок всегда отсортирован по актуальному правилу свежести.

### 15.2. Инварианты по Trainings

- каждая тренировка показывает дату, день недели и упражнения;
- одинаковые подряд идущие подходы в истории объединяются;
- заметки подходов отображаются рядом со строкой, если они есть;
- новые тренировки текущего дня должны подниматься выше более старых тренировок этого же дня.

### 15.3. Инварианты по Progress

- `Workouts` равно числу тренировок в выбранном диапазоне;
- `Volume` равно сумме `weight * reps` по всем валидным подходам диапазона;
- `Average Volume / Workout = Volume / Workouts`, если `Workouts > 0`;
- `Top Exercise` определяется по максимальному суммарному объёму;
- `Heaviest Set` определяется по максимальному весу, затем по повторениям;
- `Recent Volume Trend` содержит максимум 7 последних тренировок диапазона.

### 15.4. Инварианты по New

- в одной тренировке упражнение нельзя добавить дважды как отдельную сущность;
- `Стандарт` доступен только при наличии валидной истории;
- `Стандарт` всегда добавляет подход `12 reps`;
- стартовый вес берётся из последней подходящей тренировки;
- `set_index` после сохранения всегда последовательный и начинается с 1;
- восстановленный черновик можно сбросить одной кнопкой;
- после сохранения тренировка сразу участвует в `Trainings`, `Progress` и в логике `Стандарт`.

## 16. Ограничения текущей версии

Текущая версия не поддерживает:

- удаление тренировок;
- редактирование уже сохранённой тренировки;
- редактирование или удаление отдельного подхода;
- редактирование `notes`, `focus`, `plan_id`;
- полноценную кросс-девайс надёжную Telegram-auth во всех launch context без fallback;
- отдельный бизнес-сценарий для `deload`.
