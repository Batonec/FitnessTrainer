# Trainer Business Logic

## 1. Зачем нужен этот документ

Документ фиксирует текущее продуктовое поведение `Trainer`:

- как приложение авторизует пользователя;
- какие данные считаются источником истины;
- как устроены экраны `Trainings`, `Progress`, `Weight` и flow создания тренировки;
- какие инварианты должны сохраняться в тестах.

Это не инструкция по деплою. Технические детали окружения и запуска вынесены в [telegram_miniapp/README.md](./telegram_miniapp/README.md).

## 2. Общая модель продукта

`Trainer` работает как Telegram Mini App с backend API на SQLite.

В текущей версии:

- справочник упражнений загружается из JSON;
- тренировки и вес тела хранятся на backend;
- черновик новой тренировки и некоторые UI-предпочтения живут локально;
- одна база используется для всех пользователей, но данные изолированы по `user_id`;
- запуск из Telegram является основным продовым режимом;
- обычный браузер нужен для локальной разработки.

## 3. Режимы пользователя и сессии

### 3.1. Telegram verified user

Предпочтительный режим:

- Mini App открыт из Telegram;
- frontend передаёт `Telegram.WebApp.initData`;
- backend валидирует подпись;
- пользователь связывается с записью в таблице `users`;
- дальнейшие запросы работают в контексте этого пользователя.

### 3.2. Telegram unsafe fallback

Если signed `initData` не пришёл или не прошёл валидацию, но Telegram отдал `unsafeUser`, backend может впустить пользователя в fallback-режиме:

- пользователь сохраняется как Telegram-user;
- `auth_mode = telegram_unsafe`;
- данные всё равно изолированы по своему `user_id`.

Это совместимый деградационный путь, а не идеальный продовый сценарий.

### 3.3. Telegram recovery user

Если Mini App открыт из Telegram вообще без пригодного auth-payload, backend может использовать заранее заданного recovery-user:

- `auth_mode = telegram_recovery`;
- используется конкретный `MINIAPP_TELEGRAM_RECOVERY_USER_ID`;
- этот режим нужен, чтобы не блокировать разработку и ручные проверки.

### 3.4. Browser debug user

Для локальной отладки браузер без Telegram может автоматически получать debug-сессию:

- используется alias `browser-default`, если не задан другой;
- данные такого пользователя живут отдельно от Telegram-user;
- browser debug не должен прилипать к Telegram-shell.

## 4. Основные сущности

### 4.1. Упражнение

Упражнение из справочника содержит:

- `id`
- `name`

### 4.2. Подход

Подход содержит:

- `set_index`
- `weight`
- `reps`
- `notes`

### 4.3. Тренировка

Тренировка содержит:

- `id`
- `client_id`
- `workout_date`
- `created_at`
- `updated_at`
- `data.load_type`
- `data.exercises`

### 4.4. Запись веса тела

Запись веса тела содержит:

- `id`
- `entry_date`
- `weight`
- `created_at`
- `updated_at`

Для одного пользователя на одну дату существует только одна актуальная запись. Повторное сохранение на ту же дату обновляет значение, а не создаёт дубль.

## 5. Источники данных

### 5.1. Справочник упражнений

Источник:

- [telegram_miniapp/web/data/exercises.json](./telegram_miniapp/web/data/exercises.json)

Он используется для:

- плитки выбора упражнений в конструкторе тренировки;
- селектора упражнения на экране `Progress`.

### 5.2. Серверные тренировки

Источник истины для истории, прогресса и логики стартового веса:

- SQLite backend
- таблица тренировок текущего пользователя

Демо-история больше не подмешивается.

### 5.3. Серверные записи веса тела

Источник истины для экрана `Weight`:

- SQLite backend
- таблица `body_weights`

### 5.4. Что хранится локально

В `localStorage` хранятся только UI-состояния:

- черновик тренировки;
- текущая вкладка;
- выбранный диапазон на `Progress`;
- выбранный диапазон на `Weight`;
- выбранное упражнение на `Progress`.

Пользовательские тренировки и записи веса тела не считаются локальным источником истины.

## 6. Навигационная модель

Нижний бар сейчас содержит три вкладки:

- `Trainings`
- `Progress`
- `Weight`

Создание новой тренировки не является отдельной вкладкой.

Новая тренировка открывается:

- по FAB `+` на экране `Trainings`;
- либо через возврат в уже существующий черновик.

По умолчанию приложение открывается на `Trainings`.

## 7. Правила сортировки данных

### 7.1. Сортировка тренировок

Тренировки сортируются:

1. по `workout_date` по убыванию;
2. при совпадении даты — по `created_at` по убыванию;
3. затем по `updated_at` по убыванию;
4. затем по `id` по убыванию.

Это правило используется в:

- `Trainings`;
- логике выбора последнего рабочего веса;
- аналитике прогресса.

### 7.2. Сортировка записей веса тела

Записи веса тела для графика упорядочиваются от старых к новым.

## 8. Экран Trainings

### 8.1. Назначение

Экран показывает историю тренировок текущего пользователя.

### 8.2. Состав карточки тренировки

Карточка включает:

- дату и день недели в одной строке;
- бейдж нагрузки, если есть `load_type`;
- компактный список упражнений;
- сгруппированное summary по сетам.

### 8.3. Формат summary по сетам

Подходы группируются по последовательным одинаковым весам. Внутри такой группы повторения схлопываются в компактную строку.

Примеры:

- `60кг ×14`
- `60кг ×14×3`
- `90кг ×13×2, 12`

Если у подходов есть `notes`, они выводятся отдельной компактной строкой под упражнением.

### 8.4. Swipe actions

Карточка тренировки поддерживает свайп влево и открывает два действия:

- редактирование;
- удаление.

Карточка должна уметь закрываться назад:

- свайпом вправо;
- тапом по свободной области.

### 8.5. FAB на экране истории

Внизу справа расположен FAB:

- если черновика с сетами нет, это обычный `+`;
- если черновик существует, FAB становится точкой входа обратно в него.

Если есть живой черновик, вокруг FAB показывается кольцевой прогресс по основной плитке упражнений.

### 8.6. Next workout plan

В коде существует логика генерации `next workout plan`, но текущий UI её не показывает.

Фича сейчас отключена флагом:

- `ENABLE_NEXT_WORKOUT_PLAN = false`

## 9. Конструктор новой тренировки

### 9.1. Вход в flow

Экран `New Workout` открывается из `Trainings` по FAB.

Если в `localStorage` уже есть черновик, пользователь попадает в него, а не в пустой flow.

### 9.2. Дата тренировки

Дата редактируется из компактного чипа в верхнем баре.

Отдельной большой карточки даты на экране нет.

### 9.3. Стартовое состояние конструктора

Если пользователь открывает `New Workout` без уже сохранённых сетов, экран не должен быть пустым.

Вместо пустого состояния конструктор сразу показывает preview-карточки основных упражнений, как будто пользователь уже выбрал их из основной подборки, но ещё не добавил ни одного сета.

Редкие упражнения по-прежнему доступны отдельно через блок `Ещё упражнения`.

### 9.4. Как формируется основная подборка

Основная плитка строится по реальной истории тренировок пользователя:

- приоритет получают часто встречающиеся упражнения;
- дополнительно учитывается их типичная позиция внутри тренировки;
- целевой размер основной подборки — до 6 упражнений;
- `Жим гор.` всегда остаётся в редких упражнениях.

### 9.5. Как ведут себя preview-карточки и picker во время draft-flow

- упражнение не должно дублироваться в текущем черновике;
- основная шестёрка отображается как preview-карточки, пока у упражнения нет реального сета;
- после первого сохранённого сета preview-карточка превращается в обычную draft-карточку;
- если пользователь открыл упражнение, но закрыл модалку без сохранения первого сета, пустая карточка не должна остаться в черновике;
- `Ещё упражнения` открывает редкий каталог отдельно и не должен пополнять основную шестёрку;
- когда все упражнения основной шестёрки реально начаты, показывается completion-state `Круто, тренировка закончена`, но редкий каталог всё ещё доступен.

### 9.6. Суперсеты

Конструктор поддерживает параллельное ведение нескольких упражнений:

- не нужно “заканчивать” одно упражнение, чтобы начать второе;
- пользователь может чередовать сеты между карточками упражнений;
- у каждой карточки есть свои быстрые действия и собственная история прогресса.

### 9.7. Карточка упражнения в конструкторе

Карточка упражнения содержит:

- заголовок с названием и маленькой пиктограммой упражнения;
- компактную reference-строку `прошлый результат → план`, собранную по последней подходящей тренировке;
- компактную синюю строку текущих уже внесённых сетов, использующую ту же логику группировки, что и история тренировок;
- один большой синий `+` справа.

Тап по синей summary-строке должен редактировать именно последний сохранённый сет упражнения, а не первый.

### 9.8. Быстрое добавление сета по плану

Короткий тап по синему `+` не открывает модалку, а сразу добавляет следующий сет по плану.

Правила:

- источник плана — последняя подходящая тренировка по этому упражнению;
- очередность идёт по порядку прошлых сетов;
- в каждый повтор добавляется `+1`;
- если в истории для упражнения ничего нет, fallback остаётся простым: берётся последний известный вес и `12` повторений.

### 9.9. Ручной ввод сета по long press

Long press по синему `+` открывает ручной set-modal.

Ручной ввод не должен открываться по случайному тапу в любую область карточки. Только по long press на самой кнопке `+`.

### 9.10. Set-modal

Актуальная модалка сета устроена так:

- закрывается тапом вне модалки;
- отдельной кнопки `Отмена` нет;
- отдельного текстового заголовка `Новый сет` / `Редактировать сет` нет;
- сверху показывается блок упражнения с иконкой и reference-строкой `было → план`;
- ниже идут две компактные stepper-строки:
  - вес в формате `120,5 кг`;
  - повторы в формате `11 ПТ`;
- у stepper-кнопок используется тот же общий синий круглый стиль, что и у `+` на карточке упражнения;
- основное действие модалки только одно: `Применить`.

### 9.11. Long press по карточке упражнения

Long press по поверхности карточки упражнения открывает bottom sheet.

В нём доступны два действия:

- удалить последний сет;
- удалить всё упражнение.

Swipe-удаление у draft-карточек больше не является частью актуального UX.

### 9.12. FAB-ы внутри конструктора

Пока в черновике нет ни одного реального сета, FAB-ы не показываются.

После появления первого сета становятся видны:

- чёрный save-FAB справа;
- красный reset-FAB слева.

### 9.13. Reset-flow

Красная кнопка не сбрасывает черновик мгновенно.

Она сначала показывает confirm, и только после подтверждения:

- очищает черновик;
- возвращает экран в исходное состояние.

### 9.14. Кольцевой прогресс вокруг FAB

Save-FAB и FAB на `Trainings` могут быть окружены кольцевым прогрессом.

Правила:

- прогресс растёт только по упражнениям основной шестёрки;
- упражнение считается завершившим вклад в прогресс только после появления первого сета;
- простое открытие модалки без применения сета прогресс не увеличивает;
- кольцо анимируется вперёд и назад при изменении состояния черновика.

### 9.15. Автоскролл конструктора

Если пользователь добавляет новое упражнение и его карточка оказывается за пределами видимой области, экран должен подскроллиться так, чтобы новая карточка оставалась выше нижних FAB-ов.

### 9.16. Сохранение тренировки

При сохранении:

- frontend отправляет тренировку на backend;
- backend нормализует payload;
- тренировка получает серверный `id`;
- черновик очищается;
- новая запись сразу появляется в `Trainings`, `Progress` и влияет на быстрый planned add для следующей тренировки.
- save-FAB должен показывать явное loading-состояние, а не выглядеть как зависший UI.

### 9.17. Автоматический `load_type`

Если `load_type` не задан явно, backend выводит его из суммарного объёма:

- `heavy` — если объём `>= 3000`;
- `medium` — если объём `>= 1600` и `< 3000`;
- `light` — если объём `< 1600`.

## 10. Экран Progress

### 10.1. Назначение

Экран показывает не общую тоннажную аналитику, а быстрый ответ на два вопроса:

- сколько тренировок было в выбранном диапазоне;
- как растут рабочий вес и повторения по конкретному упражнению.

### 10.2. Диапазоны

Доступны:

- `7D`
- `30D`
- `All`

### 10.3. Summary

В summary показывается только:

- количество тренировок за период.

Старые объёмные метрики вроде `Volume`, `Top Exercise`, `Heaviest Set` и `Average Volume` больше не считаются частью текущего продукта.

### 10.4. Выбор упражнения

Селектор упражнений строится из реальной серверной истории пользователя.

Каталог может использоваться как fallback, но основной смысл экрана — анализ уже реально встречавшихся упражнений.

### 10.5. Серия на графике

Для каждой тренировки в выбранном диапазоне строятся две линии:

- `Вес` — самый тяжёлый сет упражнения на этой тренировке;
- `Повторы` — сет с максимумом повторений на этой тренировке.

### 10.6. Карточки-снимки

Под выбранное упражнение показываются три snapshot-карточки:

- `Старт`
- `Сейчас`
- `Изменение`

Они отражают первый и последний points выбранного диапазона.

## 11. Экран Weight

### 11.1. Назначение

Экран отвечает за контроль веса тела.

### 11.2. Основные элементы

Экран содержит:

- диапазоны `7D / 30D / All`;
- компактный inline composer;
- summary-strip;
- график веса тела.

### 11.3. Inline composer

Composer состоит из:

- чипа с датой;
- поля ввода веса;
- кнопки сохранения.

Надзаголовков и лишних поясняющих текстов в актуальном UI нет.

### 11.4. Правило сохранения веса

На одну дату у пользователя должна существовать только одна запись.

Поэтому повторное сохранение на ту же дату:

- не создаёт дубль;
- обновляет уже существующую запись.

### 11.5. Summary-strip

На экране показываются три компактные метрики:

- `Последний`
- `Дельта`
- `Записей`

### 11.6. График веса тела

График показывает:

- только реальные записи пользователя;
- слева по оси `Y` — только реально введённые значения веса, без искусственных промежуточных меток;
- снизу по оси `X` — даты начала и конца текущего диапазона данных.

### 11.7. Удаление записи веса

Тап по точке графика открывает confirm на удаление записи.

При подтверждении:

- запись удаляется на backend;
- график и summary сразу пересчитываются.

## 12. Правила отображения чисел

### 12.1. Рабочие веса в тренировках

Вес отображается компактно, без пробела перед `кг`.

Примеры:

- `60кг ×14`
- `60кг ×14×3`

### 12.2. Повторы

Повторы отображаются целыми числами.

### 12.3. Вес тела

Для веса тела не нужно искусственно округлять метки графика до одной цифры после запятой.

Если запись введена как точное значение, UI должен показывать её без навязанной дополнительной нормализации шкалы.

## 13. Загрузка и ошибки

### 13.1. Старт приложения

На старте приложение:

1. инициализирует Telegram shell, если он доступен;
2. восстанавливает черновик и UI-состояние;
3. резолвит пользовательскую сессию;
4. загружает справочник упражнений, тренировки и вес тела;
5. строит итоговое состояние.

### 13.2. Loading-screen

Во время старта показывается отдельный branded loading-screen.

### 13.3. Ошибка загрузки

Если boot не собрался, показывается экран ошибки с возможностью повторить загрузку.

## 14. Инварианты для тестов

### 14.1. Пользовательские данные

- данные одного пользователя не должны попадать в контекст другого;
- Telegram-shell не должен “залипать” на debug-user;
- recovery и fallback должны быть предсказуемыми и покрыты тестами.

### 14.2. Trainings

- список всегда берётся с backend;
- карточки отсортированы по правилу свежести;
- swipe edit/delete работают;
- summary сетов компактно группируется;
- новая тренировка сразу появляется в истории.

### 14.3. New Workout

- FAB-ы не появляются до первого реального сета;
- стартовый экран конструктора должен сразу показывать preview-карточки основной шестёрки, а не пустую плитку;
- пустое выбранное упражнение не должно сохраняться после закрытия модалки без применения первого сета;
- короткий тап по синему `+` должен добавлять следующий planned set, а long press по нему открывать ручной modal-flow;
- summary введённых сетов в draft-карточке должно использовать ту же логику схлопывания, что и история тренировок;
- summary-текст должен редактировать последний сет упражнения;
- long press по карточке упражнения должен открывать bottom sheet удаления;
- completion-state основной подборки должен появляться только после реального исчерпания её упражнений;
- автоскролл должен удерживать новые карточки выше FAB-ов;
- прогресс-кольцо должно расти только по упражнениям с сетами;
- при возвращении к живому черновику пользователь попадает в него, а не в новый пустой flow.

### 14.4. Progress

- число тренировок в summary равно числу тренировок в выбранном диапазоне;
- для выбранного упражнения линии строятся по лучшему весу и лучшим повторам на тренировку;
- если данных нет, экран должен показывать понятное пустое состояние.

### 14.5. Weight

- запись на ту же дату должна обновляться, а не дублироваться;
- график должен строиться по реальным данным пользователя;
- удаление по тапу на точку должно реально удалять запись на backend.

## 15. Текущие ограничения

- UI `Плана следующей тренировки` сейчас отключён, хотя код фичи в проекте сохранён;
- Telegram auth всё ещё имеет fallback/recovery-ветки, потому что launch-context Telegram не всегда стабильно отдаёт signed payload;
- одна запись веса тела на дату — это текущая продуктовая модель, а не история нескольких измерений в течение дня;
- `focus`, `notes` на уровне всей тренировки и полноценные AI-рекомендации пока не являются частью текущего UX.
